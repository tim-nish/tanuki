record — on conflict, the issue wins).

Origin: ledger finding F63 (rec=3, ledger-lifecycle). Spec decision executed
2026-07-17 ("warn-only + declare"): the lifecycle arrow path in
`docs/tanuki-spec.md` (Finding lifecycle) is the canonical operator path,
not a tool-enforced state machine — `set-status` records any state,
sequencing belongs to the command layer, and `open` → `accepted` (watching
accept) / `accepted` → `open` (regression re-open) are expected direct
transitions. The tool's job is to make a skip visible, never to block it.

As an operator driving `tanuki-ledger set-status` by hand, I want a
non-canonical transition to print a one-line notice naming the skipped
state, so that I can tell a deliberate shortcut from a sequencing mistake
without reading the tool's source.

## Acceptance criteria

- **Given** a finding in `open`, **when** I run `set-status --status
  accepted`, **then** the transition succeeds and a one-line notice on
  stderr names the skipped state (e.g. `note: open → accepted skips
  proposed — expected for a watching-list accept`); exit code and the JSON
  contract are unchanged.
- **Given** a canonical transition (`open` → `proposed`, `proposed` →
  `accepted`, `proposed` → `dismissed`), **when** I run `set-status`,
  **then** no notice is printed.
- **Given** any transition, **then** `set-status` never rejects on
  lifecycle grounds — enforcement stays with the command layer per
  `docs/tanuki-spec.md` (Finding lifecycle).
- **Given** `set-status --help`, **then** it lists the enum and names the
  canonical path, stating that transitions are recorded with a notice,
  never rejected (spec-short-command-surface D3).
- Test coverage in `tools/tests/` exercising: a canonical transition
  (silent), a skip transition (notice, success), and the re-open
  transition `accepted` → `open` (notice, success).
