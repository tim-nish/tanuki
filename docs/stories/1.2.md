Origin: ledger finding F4 (chronic, rec=3) — once `iter-verify` has closed an
iteration, an external-modification breaker leaves no working recovery path:
`rollback` requires an open iteration, so it breakers (and overwrites
`last_breaker` — ledger F17), and the only recovery was undocumented manual
`git reset` learned from reading source. Contract: spec-tanuki-loop
"Recovery from the external-modification breaker (attended)" (ADDED
2026-07-15).

As a tanuki-loop operator whose run stopped on an external-modification
breaker, I want an explicit attended `recover` subcommand with a restore mode
and an adopt mode, so that I can choose between discarding the external
change and re-baselining around it without manual git surgery — and without
ever mutating a closed iteration.

## Acceptance criteria

- **Given** a run whose `last_breaker` is an external-modification breaker
  and whose last iteration is closed, **when** I run `tanuki-loop --target
  <t> --run <r> recover --restore`, **then** the loop worktree is reset to
  `head_expected` (`reset --hard` + `clean -fd`, worktree only), the audit
  records the discarded commit range and removed paths, and a subsequent
  `iter-start` passes its head/cleanliness guards.
- **Given** the same breaker state **when** I run `recover --adopt` with a
  clean worktree, **then** `head_expected` becomes the current worktree HEAD,
  the audit records old → new and the adopted range (with an explicit
  non-ancestor warning when history was rewritten rather than extended), and
  a subsequent `iter-start` passes.
- **Given** any other state (no breaker, a different breaker reason, an open
  iteration, or `--adopt` on a dirty tree), **when** `recover` is invoked,
  **then** it refuses with the breaker convention (exit 3, `{"breaker": …}`)
  and writes nothing.
- **Given** a completed recovery, **then** no recorded iteration's
  `start`/`end`/`phase` changed (closed iterations are immutable), and
  `last_breaker` reflects the recovery in the audit rather than being
  silently overwritten.
- **Given** the recover path exists, **then** `recover` requires exactly one
  of `--restore`/`--adopt` (no default), and the dashboard's breaker NEXT
  line for an external-modification stop names `recover` as the next action.
- Test coverage in `tools/tests/` exercising: restore, adopt,
  adopt-on-dirty-tree refusal, wrong-breaker refusal, open-iteration refusal,
  and iteration-immutability.
