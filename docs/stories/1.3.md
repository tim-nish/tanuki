Issue: https://github.com/tim-nish/tanuki/issues/69 (canonical discussion
record — on conflict, the issue wins).

Origin: ledger finding F42 (rec=2, loop-iteration-bracket +
morning-gate-walkthrough). `state.json` holds operator-critical fields
(`base_sha` for the gate diff, `last_breaker` for diagnosing a stop). A
partial fix landed: `tanuki-loop status` now carries `last_breaker` and
`base_sha`, but the operator still has to infer the run's overall condition
— there is no explicit healthy-vs-stopped verdict, and the `finished`
terminal state is not represented at all.

As a tanuki-loop operator returning to a run I did not just drive, I want
`status` to state an explicit run verdict — running-healthy, stopped (with
the breaker), or finished (with the reason) — so that I can tell the run's
condition from one line without cross-reading `last_breaker`, iteration
counters, or `loop/<run>/state.json`.

## Acceptance criteria

- **Given** a run with an open, breaker-free iteration bracket, **when** I
  run `tanuki-loop --target <t> status`, **then** the output leads with an
  explicit healthy/running verdict line (not merely the absence of a breaker
  field).
- **Given** a run whose last recorded event is a breaker, **when** I run
  `status`, **then** the verdict line says the run is stopped and names the
  breaker reason alongside the existing `last_breaker` detail.
- **Given** a run closed by `finish` (any reason: converged, cap, aborted,
  gate-ratified), **when** I run `status`, **then** the verdict line reports
  the run as finished with its reason — the terminal state is first-class,
  not indistinguishable from a stopped or idle run.
- **Given** any of the three verdicts, **then** `base_sha` and
  `last_breaker` remain available in the same output (no regression of the
  partial fix), and the JSON machine contract gains the verdict as a typed
  field rather than prose only.
- Test coverage in `tools/tests/` exercising all three verdicts: healthy
  mid-bracket, stopped-on-breaker, and finished.
