Issue: https://github.com/tim-nish/tanuki/issues/71 (canonical discussion
record — on conflict, the issue wins).

Origin: ledger finding F64 (rec=2, morning-gate-walkthrough +
loop-iteration-bracket). There is no `tanuki-loop commit` subcommand and no
guard against sweeping build artifacts (`__pycache__/`, `*.pyc`) into the
integration branch. Once committed, the artifacts trip `iter-verify`'s
worktree-not-clean breaker on a *later* iteration — far from the commit that
introduced them — and the breaker names neither the offending files nor the
root cause.

As a tanuki-loop operator committing work inside a loop worktree, I want the
loop to surface build-artifact contamination at the point it is introduced,
so that a stray `__pycache__` never becomes a confusing not-clean breaker
several iterations downstream.

## Acceptance criteria

- **Given** a loop worktree containing untracked build artifacts (at minimum
  `__pycache__/` and `*.pyc`), **when** the iteration reaches its verify
  step, **then** the artifacts are either excluded from consideration
  (ignore-list) or flagged by name at that step — they never survive into
  the integration branch unnoticed.
- **Given** a commit in the loop worktree that already swept artifacts in,
  **when** the next guarded step runs (`iter-verify` or an added preflight),
  **then** the diagnostic names the offending paths and states the root
  cause (committed build artifacts), not just "worktree not clean".
- **Given** the chosen design (ignore-list, preflight warning, or a
  `commit` subcommand — implementer picks and documents one), **then** the
  choice is stated in the loop docs/`--help` so operators know the guard
  exists and what it covers.
- **Given** a clean worktree with no artifacts, **then** the guard adds no
  new friction: existing bracket flows (`iter-start` → work → `iter-verify`
  → `record-cycle`) pass unchanged.
- Test coverage in `tools/tests/` exercising: artifacts present at verify,
  artifacts already committed, and the clean-path no-regression case.
