Issue: https://github.com/tim-nish/tanuki/issues/72 (canonical discussion
record — on conflict, the issue wins).

Origin: `specs/spec-tanuki-solve/SPEC.md` made the #68/#70 incident class
(contradictory proposals filed as separate issues) unrepresentable in
`/tanuki-solve`, but the brief decision pass still walked the raw finding
list with no consolidation, so any sitting deciding through `--brief` or the
end of a normal run could reproduce it. Spec decision executed 2026-07-17
("fold into brief"): the resolution rule already recorded in
spec-tanuki-solve's non-goals met its trigger, so consolidation moves into
the decision pass itself rather than being maintained as a second contract.
`specs/spec-short-command-surface/SPEC.md` D1 now requires the stage and the
ledger-anchored entry; spec-tanuki-solve D1/D3 remain the normative
taxonomy, D2 is untouched. `docs/tanuki-spec.md` is amended to match.

**Entry-point spelling is not this story's to choose.** The pass is
ledger-anchored, so its entry names the workflow (`decide`), not the artifact
(`brief`) — but the word-vs-flag shape is ratified surface-wide by D6 and
delivered by **story 1.8 (issue #74)**, which this story depends on. Land 1.8
first; this story folds behavior into whatever 1.8 spelled, and never renames
an entry point a second time.

Retirement of `/tanuki-solve` is **conditional on no capability loss**: the
original user-facing promise — one attended command from open findings that
need human judgment to labeled issues, with no raw ledger invocations — must
remain wholly reachable through the decision pass. If it is not, the command
stays.

As an operator finishing a sitting, I want every decision path — the decision
pass and the end of a normal run alike — to consolidate before it asks, so
that two findings with contradictory proposals reach one arbitration question
instead of becoming two issues I reconcile by hand.

## Acceptance criteria

- **Given** a target whose ledger holds two proposals with contradictory
  fixes for the same behavior, **when** I run `/tanuki <t> decide`, **then**
  they are presented as ONE multi-outcome conflict question naming each
  branch, and under every possible answer at most one issue is filed for
  that behavior (`spec-tanuki-solve` D4, now applied to this path).
- **Given** open findings and **no recent brief**, **when** I run
  `/tanuki <t> decide`, **then** the pass orients off `status`/`next` and
  decides whatever the ledger holds decidable — a brief is context when
  present, never a precondition (amended short-command-surface D1).
- **Given** `/tanuki <t> --brief`, **then** it reprints the latest brief and
  continues into the same pass — one code path, not a second behavior; where
  no brief exists it says so and proceeds rather than erroring.
- **Given** fewer than two candidate items, **when** the pass runs, **then**
  the consolidation stage is skipped and the minimal path is unchanged — no
  clustering work a one-finding sitting cannot need.
- **Given** a merge group accepted at the gate, **then** every constituent
  finding gets the same disposition and issue URL, and no ledger entry is
  rewritten or deleted (`spec-tanuki-solve` D1, no history surgery).
- **Given** the end of a normal run, **then** its decision pass consolidates
  identically to `decide` — one contract, not two behaviors.
- **Given** the fold has landed, **when** the capability check passes (open
  findings → labeled issues in one command, zero raw `tanuki-ledger`/`gh`
  invocations typed by hand), **then** `commands/tanuki-solve.md` is retired
  and any reference to it points at the decision pass. **If the check
  fails**, the retirement is not performed and the gap is reported — the
  one-command workflow outranks the command count.
- Existing `tools/tests/test-ledger-consolidate` coverage still passes
  unchanged — D2 is not in scope for this story.
